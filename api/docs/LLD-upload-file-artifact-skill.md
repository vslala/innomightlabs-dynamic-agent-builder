# Low Level Design: Upload File Artifact Skill

Date: 2026-06-29  
Status: Draft  
Owner: InnomightLabs API

## Summary

Add a generic first-party skill that lets agents and automations save generated text artifacts to S3-backed user artifacts, then return stable artifact metadata plus download/view links.

The skill is intentionally generic. It is not League-specific and does not generate content. The agent supplies the content, artifact type, title, filename, and optional MIME type. The skill validates the content, saves it through the artifact system, and returns the created artifact.

Use cases:

- Save an HTML report generated by an agent.
- Save CSV analysis output.
- Save Markdown notes.
- Save JSON exports.
- Save source-code text such as `.py`, `.java`, `.ts`, `.sql`, etc.

V1 is limited to text-friendly artifacts. It supports base64 input for transport safety, but the decoded payload must still fit the V1 text-artifact contract and size limit.

## Naming

Use names from the LLM/agent perspective:

- Folder: `api/src/skills/upload_file/`
- Skill id: `upload_file`
- Namespace: `core.artifacts`
- Display name: `Upload File`
- Primary action: `upload_file`
- Aliases: `save_artifact`, `upload_artifact`, `save_file`

Reasoning: agents naturally think "upload/save this file" more than "artifact writer". The artifact terminology remains in response fields and storage metadata.

## Current Architecture Fit

The platform already has the required durable storage path:

- `ArtifactService.create_artifact(...)` creates metadata, stores the body in S3, and returns `ArtifactResponse`.
- Artifact records are owned by user email, not skill id, so files remain accessible if the skill is uninstalled.
- `ArtifactSource` can capture `skill_id`, `agent_id`, `conversation_id`, automation ids, and arbitrary metadata.
- `html_report` already receives a dashboard artifact page URL as `view_url`.

This skill should call `ArtifactService` directly. It should not create a new upload API or bypass the artifact repository.

## Artifact Type Expansion

Current type:

```python
ArtifactType = Literal["html_report", "image", "file"]
```

Expand it, still keeping the set focused on text artifacts:

```python
ArtifactType = Literal[
    "html_report",
    "image",
    "file",
    "csv",
    "markdown",
    "json",
    "text",
    "code",
]
```

Notes:

- Keep `file` for backwards compatibility and generic fallback.
- Use `code` for text source files across extensions such as `.py`, `.java`, `.ts`, `.tsx`, `.sql`, `.go`, `.rs`, `.rb`, `.php`, etc.
- Do not add binary office document types in V1. The user mentioned Excel and PowerPoint as a future direction, but the agreed V1 scope is text files only.
- Existing image artifacts remain valid but this skill should not accept image generation use cases in V1.

## Browser View Behavior

Today `ArtifactService.view_url(...)` only allows `html_report`, and `ArtifactResponse.view_url` points to the SPA artifact page for HTML reports.

Change viewability to a central helper:

```python
VIEWABLE_ARTIFACT_TYPES = {"html_report", "csv", "markdown", "json", "text", "code"}

def is_browser_viewable_artifact(artifact: Artifact) -> bool:
    return artifact.artifact_type in VIEWABLE_ARTIFACT_TYPES
```

For these artifact types:

- `ArtifactResponse.view_url` should point to `/dashboard/artifacts/{artifact_id}`, same as current HTML reports.
- `GET /artifacts/{artifact_id}/view` should return an inline presigned URL with the artifact MIME type.

For unsupported/non-viewable artifacts:

- Keep returning `400 Artifact is not browser-viewable`.

Download behavior remains unchanged:

- `ArtifactResponse.url` and `/download` use `Content-Disposition: attachment`.
- `/view` uses `Content-Disposition: inline`.

This matches the current League report behavior and lets the browser decide whether it can render the content inline or prompt for download.

## Request Model

`api/src/skills/upload_file/models.py`:

```python
MAX_ARTIFACT_UPLOAD_BYTES = 10 * 1024 * 1024
UPLOADABLE_ARTIFACT_TYPES = {"html_report", "csv", "markdown", "json", "text", "code", "file"}

class UploadFileRequest(BaseModel):
    artifact_type: ArtifactType
    title: str
    filename: str
    mime_type: str | None = None
    text_content: str | None = None
    base64_content: str | None = None
    metadata: dict[str, Any] = Field(default_factory=dict)

    @model_validator(mode="after")
    def normalize(self) -> "UploadFileRequest":
        ...
```

Validation rules:

- `artifact_type` must be one of V1 uploadable text types.
- `title` and `filename` are required after trimming.
- Exactly one of `text_content` or `base64_content` must be provided.
- Decoded body must be `<= 10 MB`.
- Base64 must be valid.
- For V1, decoded base64 should be UTF-8 decodable for text artifacts. This prevents accidental binary XLSX/PPTX/PDF storage through this skill before the validator/storage contract is ready.
- `metadata` must be JSON-like and bounded. Recommended max serialized metadata length: 8 KB.

## MIME Inference

Add a small helper in the skill, not globally yet:

```python
def infer_mime_type(artifact_type: ArtifactType, filename: str, override: str | None) -> str:
    if override:
        return override.strip()
    if artifact_type == "html_report":
        return "text/html; charset=utf-8"
    if artifact_type == "csv":
        return "text/csv; charset=utf-8"
    if artifact_type == "markdown":
        return "text/markdown; charset=utf-8"
    if artifact_type == "json":
        return "application/json; charset=utf-8"
    if artifact_type == "code":
        return _code_mime_from_extension(filename)
    return mimetypes.guess_type(filename)[0] or "text/plain; charset=utf-8"
```

For source code, MIME can remain simple:

- `.py`: `text/x-python; charset=utf-8`
- `.java`: `text/x-java-source; charset=utf-8`
- `.js`, `.ts`, `.tsx`, `.jsx`: `text/javascript; charset=utf-8` or `text/plain; charset=utf-8`
- `.sql`: `application/sql; charset=utf-8`
- default: `text/plain; charset=utf-8`

Do not overfit MIME mapping in V1.

## Validator Strategy

Use a small strategy pattern so later file types can add validators without changing the action orchestration.

Recommended file:

`api/src/skills/upload_file/validators.py`

```python
class ArtifactContentValidator(Protocol):
    def validate(self, request: UploadFileRequest, body: bytes) -> None:
        ...

class HtmlArtifactValidator:
    def validate(self, request: UploadFileRequest, body: bytes) -> None:
        html = body.decode("utf-8")
        validate_safe_report_html(extract_html_document(html))

class NoopArtifactValidator:
    def validate(self, request: UploadFileRequest, body: bytes) -> None:
        return None

class ArtifactValidatorRegistry:
    def __init__(self):
        self.validators = {
            "html_report": HtmlArtifactValidator(),
        }

    def validator_for(self, artifact_type: ArtifactType) -> ArtifactContentValidator:
        return self.validators.get(artifact_type, NoopArtifactValidator())
```

HTML validation:

- Reuse `src.skills.league_insights_report.html_safety`.
- Store the extracted HTML document, not the original fenced text, if extraction changes it.
- Reject unsafe HTML with `ValueError`.

Other V1 text types:

- Use no-op validator after basic size/UTF-8 validation.
- Later add CSV/JSON/Markdown/code validators if needed.

## Action Flow

`api/src/skills/upload_file/actions.py`:

1. Validate `UploadFileRequest`.
2. Decode body through `request.to_body_bytes()`.
3. Infer MIME type.
4. Run validator strategy.
5. Create artifact:

```python
artifact = ArtifactService().create_artifact(
    owner_email=owner_email,
    artifact_type=request.artifact_type,
    title=request.title,
    filename=request.filename,
    mime_type=mime_type,
    body=validated_body,
    source=ArtifactSource(
        skill_id="upload_file",
        agent_id=context.get("agent_id"),
        automation_id=context.get("automation_id"),
        automation_run_id=context.get("automation_run_id"),
        automation_node_id=context.get("automation_node_id"),
        conversation_id=context.get("conversation_id"),
        message_id=context.get("user_message_id"),
        metadata=request.metadata,
    ),
)
```

6. Return compact result:

```python
{
    "ok": true,
    "artifact_id": "...",
    "artifact_type": "markdown",
    "title": "Build Notes",
    "filename": "build-notes.md",
    "mime_type": "text/markdown; charset=utf-8",
    "size_bytes": 1240,
    "url": "...",
    "view_url": "...",
    "download_url": "..."
}
```

For compatibility with the League report skill, return `url` as the browser page URL when `view_url` exists, and include `download_url` separately.

## Security and Safety

- Do not allow files over 10 MB.
- Do not persist arbitrary binary office files in V1.
- Sanitize filenames through existing `ArtifactStorage.sanitize_filename`.
- HTML must pass the existing safety validator before storage.
- Do not log content bodies.
- Keep returned payload compact; do not return the file content.
- Store under user-owned artifact partition, independent of skill installation.

- Manifest exposes `upload_file` with aliases.
- Text upload creates an artifact and returns download/view fields.

- MIME type is inferred from artifact type and filename.
- Explicit MIME type override wins.
